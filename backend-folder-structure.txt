Feature/Module Breakdown

Auth Module
Responsible for:
- user registration and login
- password hashing and verification
- JWT generation
- role assignment at registration (defaults to Student)

Main files:
- Features/Auth/AuthEndpoints.cs
- Features/Auth/LoginDto.cs
- Features/Auth/RegisterDto.cs
- Infrastructure/SecurityHelper.cs

Dependencies:
- `AppDbContext` (Users table)
- BCrypt.Net for password hashing
- Microsoft.IdentityModel/Tokens for JWT
- Jwt configuration in `appsettings.json`

Notes:
- Login returns a JWT via SecurityHelper.GenerateToken which embeds `ClaimTypes.NameIdentifier`, `ClaimTypes.Name`, and `ClaimTypes.Role`.
- Authorization policies are defined in `SecurityExtensions.AddJwtAuthentication()` (AdminOnly, StudentOnly, StaffOrAdmin).

Issues Module
Responsible for:
- students reporting issues with optional image upload
- staff viewing and updating assigned issues
- admins listing, assigning, and deleting issues
- caching admin issue lists in Redis

Main files:
- Features/Issues/IssueEndpoints.cs
- Features/Issues/CreateIssueDto.cs
- Features/Issues/CreateIssueValidator.cs
- Features/Issues/IssueResponseDto.cs
- Features/Issues/UpdateStatusDto.cs
- Features/Issues/UpdateStatusValidator.cs

Dependencies:
- `AppDbContext` (Issues and Users)
- `IStorageService` (LocalStorageService) for image handling
- `IConnectionMultiplexer` (StackExchange.Redis) for caching and cache invalidation
- FluentValidation for DTO validation

Behavior highlights:
- Student creates issue via `/issues/report` (multipart form). Validation occurs via `IValidator<CreateIssueDto>`.
- Image validation enforces MIME type and 5MB size, saved as `{issue.Id}.jpeg`.
- Admin endpoints under `/admin/*` are protected by `AdminOnly` policy.
- Staff endpoints under `/staff/*` use `StaffOrAdmin` policy and include ownership checks (issue.AssignedToId == userId) for status updates.
- Admin cache keys are stored as `admin:issues:{status}:{page}:{pageSize}` and invalidated on write operations.

Users Module
Responsible for:
- admin-level user listing

Main files:
- Features/Users/UserEndpoints.cs
- Features/Users/UserResponseDto.cs

Dependencies:
- `AppDbContext` (Users)
- AdminOnly authorization policy

Authentication & Authorization Architecture

JWT handling:
- `SecurityHelper.GenerateToken(User, IConfiguration)` builds a JWT with NameIdentifier, Name (email), and Role claim and sets 2 hour expiry.
- JWT signing uses symmetric key configured at `Jwt:Key` in `appsettings.json`.
- `SecurityExtensions.AddJwtAuthentication()` configures JwtBearer authentication and registers authorization policies.

Middleware flow for auth:
- `app.UseAuthentication()` runs before endpoint dispatch; JwtBearer validates tokens and populates User claims.
- `app.UseAuthorization()` enforces policies; feature endpoints call `.RequireAuthorization("PolicyName")` where necessary.

Role policies and checks:
- Policies: `AdminOnly`, `StudentOnly`, `StaffOrAdmin` are registered; implemented as `RequireRole("Admin")`, `RequireRole("Student")`, `RequireRole("Admin","Staff")`.
- Additional ownership checks exist in endpoints (e.g., staff updating status checks `issue.AssignedToId == userId`).

Permission flow:
- Authentication establishes identity & role.
- Authorization via policy reduces surface area (group routes like `/admin` and `/staff`).
- Endpoints perform final resource-level checks (e.g., forbids when assigned user mismatch).


Database Architecture

DbContext:
- `Infrastructure/AppDbContext.cs` exposes `DbSet<User>` and `DbSet<Issue>`.
- OnModelCreating configures:
  - Issue -> User relationship (HasOne User, WithMany)
  - Index on `Issue.Status`
  - Enum conversion for `Issue.Status` stored as string

Entities:
- `User`: Id, Name, Email, PasswordHash, Role
- `Issue`: Id, Title, Description, Status (IssueStatus enum), UserId, AssignedToId, Block, RoomNumber, CreatedAt, ImagePath, IsDeleted

Migrations:
- Migration files exist in `Migrations/` confirming the schema evolution including adding `ImagePath` for Issue.

Repositories & data access:
- Data access is implemented directly in endpoints using `AppDbContext` (no separate repository layer present).
- Queries use EF Core `Include()` and standard LINQ.

Seeders:
- `DatabaseExtensions.SeedDatabase(IApplicationBuilder)` seeds default Admin and Staff users if missing.


Shared Infrastructure

Caching:
- Redis `IConnectionMultiplexer` is registered in `DatabaseExtensions.AddRedisCache()`.
- Used in Issues admin listing for caching query results for 5 minutes.
- Cache invalidation implemented via scanning keys matching `admin:issues:*` and deleting them.

File uploads:
- `IStorageService` abstraction with `LocalStorageService` implementation.
- Files saved under `uploads/` and served as static files via `app.UseStaticFiles()` mapped to `/uploads`.
- Image URL resolved by `LocalStorageService.GetUrl(fileName, HttpContext)`.
- Upload validation (MIME/size) is done inline in the `/issues/report` endpoint.

Logging:
- Serilog configured in `Program.cs` to log to console and `logs/log.txt`.
- `GlobalExceptionMiddleware` logs unhandled exceptions using Serilog.

Configuration:
- Configuration via `appsettings.json` and environment variables; `Program.cs` reads `PORT` and `Jwt:Key`, `DefaultConnection`, `Redis:ConnectionString`.

Helpers & Utilities:
- `SecurityHelper` centralizes token generation and password hashing/verification.
- `EnumSchemaFilter` converts enums to strings for Swagger.


Dependency Injection Overview

Registered services (not exhaustive but primary):
- `services.AddValidatorsFromAssemblyContaining<Program>()` (FluentValidation validators)
- `IStorageService` -> `LocalStorageService` (singleton)
- `AppDbContext` via `AddDbContext<AppDbContext>` (scoped by EF Core per request)
- `IConnectionMultiplexer` (singleton)
- Authentication and Authorization (standard lifetimes inside the framework)
- RateLimiter configured with named limiters (`login`, `issue`)

Patterns:
- Minimal APIs pattern: endpoints are declared in static `Map*Endpoints` extension methods.
- Services wiring: `AddAppServices()` centralizes app-level registrations.
- Validators are auto-registered from assembly.


Middleware Pipeline

Order and purpose (as configured in `Program.cs`):
1. `UseSwaggerDocumentation()` - swagger + UI
2. `UseCors("AllowAll")` - open CORS policy
3. `UseRateLimiter()` - named rate limiters (applied per-endpoint with `.RequireRateLimiting`)
4. Static files middleware for `/uploads`
5. `UseSerilogRequestLogging()` - request logging
6. `UseGlobalExceptionHandler()` - centralized exception handling
7. `UseAuthentication()` - JWT validation
8. `UseAuthorization()` - policy enforcement
9. Endpoint mapping and execution

Notes:
- Rate limiting applied to routes using `.RequireRateLimiting("login")` and `.RequireRateLimiting("issue")`.
- No explicit rate limiting middleware ordering issues observed; `UseRateLimiter()` is invoked before auth.

AI AGENT NOTES

- Business logic location: Mostly inside the `Features/*/ *Endpoints.cs` minimal API files. These contain DB interactions, validation orchestration, file handling, and caching.
- Validation location: FluentValidation validators are in `Features/Issues` (CreateIssueDto validator, UpdateStatus validator) and auto-registered via `AddValidatorsFromAssemblyContaining<Program>()`.
- Auth logic location: `Infrastructure/SecurityHelper.cs` (token creation + password), `Infrastructure/Extensions/SecurityExtensions.cs` (auth config). `RequireAuthorization()` in endpoints enforces per-route policy.
- Role checks: Per-route policies (`AdminOnly`, `StudentOnly`, `StaffOrAdmin`) plus resource checks (e.g., `issue.AssignedToId != userId` leads to `Results.Forbid()`).
- API responses shaped: Endpoints map EF entities into `*ResponseDto` classes (`IssueResponseDto`, `UserResponseDto`) before returning.
- File upload handling: Done inside `POST /issues/report` in `IssueEndpoints.cs` using `IStorageService` (LocalStorageService). File validation (MIME and max size 5MB) enforced inline.
- Database interactions: Performed directly using `AppDbContext` in endpoints; no repository abstraction.
- Separation of concerns: Minimal API style puts routes, validation orchestration, and business logic together â€” moderate mixing of concerns; light service abstraction (Storage) exists.


Uncertainties & Notes

- No explicit repository layer or background job system detected. (Search for `BackgroundService`, `IHostedService`, `Hangfire` returned no project references.)
- No WebSocket endpoints or real-time notifications implemented. (Repo contains no WebSocket code; `.vs` files reference IIS WebSocket features but nothing in app code.)
- Caching limited to admin issues listing via Redis; no global cache abstraction found.
- Rate limiter is configured with named fixed-window limiters; their use is per-endpoint via `.RequireRateLimiting`.


Actionable next steps for developers or agents
- To add a repository layer: extract EF access from `*Endpoints.cs` into services under `Infrastructure/Services` or `Features/*/Services`.
- To add background processing: implement `IHostedService` for periodic jobs and register it in `Program.cs`.
- To support cloud storage: add alternative `IStorageService` implementations (S3, Azure Blob) and configure via DI.
